AIRO (AI Research Orchestrator) - Full Architecture Report
=============================================================

1. WHAT DOES THE PROJECT DO?
-------------------------------------------------------------
AIRO is an end-to-end, multi-agent AI system that automates the entire machine learning experimentation lifecycle. 
When given a raw tabular dataset, AIRO acts as an autonomous Senior Data Science team. It cleans the data, 
programmatically brainstorms suitable machine learning algorithms, trains dozens of model configurations concurrently, 
rigorously audits them for common pitfalls (like overfitting or data leakage), and finally outputs a beautiful, 
production-ready PDF report complete with SHAP explainability charts. 

2. WHY WAS IT BUILT? (THE PROBLEM)
-------------------------------------------------------------
The standard data science workflow—cleaning data, writing boilerplate scikit-learn code, selecting algorithms, 
tuning hyperparameters, running cross-validation, and compiling findings into a PowerPoint or PDF—is highly repetitive, 
tedious, and prone to human error. Practitioners often spend days executing trivial baselines. Furthermore, 
junior data scientists often lack the intuition to catch subtle algorithmic anomalies like target leakage or 
aggressive overfitting on small datasets.

3. WHY IS IT NECESSARY? (THE SOLUTION)
-------------------------------------------------------------
AIRO removes the manual friction of initial ML prototyping. By orchestrating six specialized LLM-backed and procedural agents, 
the system enforces rigorous, standardized methodologies uniformly without human fatigue. 
It is necessary because it allows teams and researchers to go from a raw, unknown dataset to a fully documented, 
statistically sound baseline in under 5 minutes. It serves as an incredibly powerful "first pass" tool that tells a 
researcher exactly what models perform best and why, saving countless hours of boilerplate coding.

4. COMPREHENSIVE FOLDER & FILE BREAKDOWN
-------------------------------------------------------------

[ORCHESTRATOR] - The "Brain" of the system
The orchestrator directory handles the LangGraph state machine. It defines how agents pass information to one another.
• orchestrator/graph.py : 
  The core routing logic. It defines the Nodes (Agents) and Edges (transitions), creating the actual flowchart that the execution follows.
• orchestrator/router.py : 
  Contains conditional logic. For example, if the Data Agent hits an unrecoverable error, the router stops the graph early rather than crashing.
• orchestrator/state.py : 
  Defines `AIROState`, the single source of truth (dataclass) that gets passed from agent to agent. It acts as the shared memory for the bots.
• orchestrator/runner.py : 
  The Command Line Interface (CLI) entrypoint.

[AGENTS] - The Specialized Workforce
These files represent the individual "employees" of the system.
• agents/data_agent.py (The Cleaner) : 
  Cleans incoming datasets, handles NaN imputation, applies one-hot/label encoding, and creates stratified train/val/test splits.
• agents/config_agent.py (The Architect) : 
  Sends the dataset metadata to a Groq LLaMA model so the LLM can intelligently brainstorm hyperparameter grids based on the specific problem.
• agents/training_agent.py (The Engine) : 
  A procedural workhorse. Uses a ThreadPoolExecutor to train dozens of models in parallel based on the Config Agent's ideas.
• agents/critic_agent.py (The Reviewer) : 
  Audits trained models. It checks train vs validation metrics to flag overfitting and enforces statistical rigour. If a model is bad, the Critic fails it.
• agents/evaluator_agent.py (The Judge) : 
  Compares all surviving models against a naive "dummy" baseline to score actual "lift" or improvement, ultimately ranking them on a Leaderboard.
• agents/reporter_agent.py (The Analyst) : 
  Compiles the final outputs, injects them into a Jinja2 template, and converts them to Markdown and a WeasyPrint PDF.

[FRONTEND & API] - The User Interface
The GUI was completely migrated to a high-performance Next.js 14 App Router stack with a FastAPI backend, providing a robust, non-blocking experience.
• api/main.py : 
  The FastAPI ASGI server. Acts as a bridge between the LangGraph orchestrator and the Next.js client, utilizing background tasks and Server-Sent Events (SSE).
• airo-frontend/app/ : 
  The Next.js 14 routing structure for the dashboard pages (Home, Run Experiment, Live Trace, Leaderboard, Report Viewer).
• airo-frontend/components/ : 
  Modular React components built with Shadcn UI, Framer Motion, and Tailwind CSS v4 to enforce a premium cyan/white visual language.
• airo-frontend/store/experimentStore.ts : 
  Zustand global state manager for coordinating SSE streaming logs and experiment data across the frontend.

[TOOLS] - The Utility Belt
Reusable backend functions that power the agents.
• tools/data_tools.py : 
  Under-the-hood Pandas and Scikit-learn data wrangling logic called by the Data Agent.
• tools/llm_tools.py : 
  Wrappers for communicating with the Groq/Gemini APIs securely, including retry and JSON parsing logic.
• tools/metrics_tools.py : 
  Standardizes how classification and regression metrics (Accuracy, F1, RMSE) are calculated across all models.
• tools/mlflow_tools.py : 
  Handles logging runs, parameters, and metrics to the local SQLite MLflow tracking server.
• tools/shap_tools.py : 
  Generates "Feature Importance" plots that explain mathematically why models are making their predictions.
• tools/report_tools.py : 
  The HTML/CSS templates and Jinja2 rendering logic used by the Reporter Agent to write the final PDF.

[STORAGE DIRECTORIES] - Data persistence
• data/raw/ : Where user-uploaded CSV/Parquet files first enter the system.
• data/processed/ : Where cleaned artifacts are stored.
• data/splits/ : Where Train/Val/Test subsets are temporarily cached for models to read.
• models/ : Where winning model weights (`.pkl` files) are serialized and saved.
• reports/ : Where the final PDF and Markdown summaries are exported.

[CONFIGURATION]
• requirements.txt : Python dependencies required to run the project.
• .env : Environment variables (API keys, performance flags).
• README.md : The front-page documentation for standard GitHub viewers.
